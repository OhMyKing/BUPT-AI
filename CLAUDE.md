# BUPT-AI Agent 整理协议

本文件供 Codex、Claude Code、OpenClaw、Hermes 等具备文件与 Git 操作能力的 Agent 阅读。无论 Agent 是否原生识别 `CLAUDE.md`，贡献者都可以在提示词中要求其先阅读本文件。

## 目标与边界

将贡献者指定的本地课程资料整理进 BUPT-AI，并生成便于人类审核的变更。Agent 必须：

- 只处理用户明确指定的文件、文件夹和本仓库；
- 保留源目录中的原始文件，仅复制到仓库后整理；
- 先盘点、分类和提审，再执行批量复制、删除、合并、提交或推送；
- 不因整理任务擅自发布资料、创建 PR、强制推送或改写历史；
- 遇到课程归属、学期、版权、隐私或删除选择不确定时暂停并请用户确认。

## 标准工作流

### 1. 读取仓库规则

开始前阅读：

- `README.md`；
- `docs/CONTRIBUTING.md`；
- `.gitignore`；
- `.gitattributes`；
- 目标课程或项目已有的 README。

检查 Git 工作树是否已有未提交变更。已有变更默认属于用户，不得覆盖、还原或混入本次提交；无法隔离时先提审。

### 2. 盘点源资料

递归检查用户给出的绝对路径，包括压缩包、文档、图片、视频和代码。压缩包应解压到仓库外的临时目录检查，不直接把解压过程写入仓库。

建立外置清单，至少记录：

- 原始绝对路径；
- 推断的学期、课程与资源类型；
- 建议目标路径；
- 文件大小、格式和哈希；
- 重复项、多版本及建议保留方式；
- 凭证、隐私、模型、数据集、构建产物和版权风险；
- 需要用户决定的问题。

临时清单不得提交。可使用仓库已忽略的 `RESOURCE_PLAN.md`，或使用仓库外临时文件。

### 3. 规划目录

目录遵循：

```text
YY_fall或YY_spring/课程中文名/资源类型/可选的贡献者或项目目录/
```

资源类型按实际内容选择，如 `教材`、`课件`、`作业`、`实验`、`考题`、`课设`、`论文`、`复习资料`。不要创建空目录，也不要仅凭文件名中的“期末”判断类型：考点梳理属于复习资料，项目期末报告仍属于课设。

向已有目录贡献另一个版本时，可以用姓名、用户名或项目名建立子目录，避免覆盖现有资料。

### 4. 第一次提审：整理方案

执行文件变更前向用户展示：

- 源文件总数和各类型数量；
- 每门课程的目标目录；
- 将复制、合并、改名、排除的内容；
- 重复、多版本和冲突处理；
- 隐私、凭证、版权和大文件风险；
- 需要确认的具体选择。

获得明确同意后才能继续。不得删除源目录中的任何文件。

### 5. 安全清理

复制到仓库的内容必须检查并处理：

- API Key、Token、密码、私钥、Cookie、连接串和真实 `.env`；
- 身份证件、签名、家庭住址、私人联系方式、精确位置和不必要的文档元数据；
- 训练模型、权重、公开数据集及其缓存；
- `node_modules`、虚拟环境、IDE 配置、系统文件、日志、临时文件和编译产物；
- 空文件、损坏文件及无意义副本。

学号、真实姓名可以作为课程归档信息保留，但仍应删除与课程贡献无关的高风险个人材料。公开数据集和模型应在项目 README 中写明官方来源、版本和获取方法，不得把实体文件提交进仓库。凭证改为环境变量，并提供只有无效占位符的 `.env.example`。

Office、PDF、图片和视频不能只按扩展名检查：应尽可能检查可提取文字、内嵌对象、作者元数据、EXIF/GPS、画面水印、字幕和音轨。无法可靠自动检查时必须在最终提审中明确列出，交由用户人工确认。

### 6. 项目可读性

独立课设或代码项目缺少 README 时，阅读代码后补充：

- 项目目的与课程背景；
- 目录和主要模块；
- 环境、依赖、配置和运行步骤；
- 已移除模型、数据集或外部服务的下载方式；
- 演示视频、报告和 PPT 的位置；
- 已知限制和复现条件。

不要捏造无法从项目确认的运行结果、版本或下载地址。

### 7. 大文件与 Git LFS

提交前列出文件大小，并遵循：

- 普通 Git 对象必须小于 GitHub 的 100 MiB 硬限制；
- 接近或超过 50 MiB 的二进制文件优先评估 Git LFS；
- 演示视频以及确有保留必要的大型 PDF、PPT 等使用 Git LFS；
- 模型和公开数据集即使可用 LFS，也不得提交；
- 先确认贡献者和仓库具有足够的 LFS 存储与带宽；
- 用 `git check-attr filter -- <文件>` 和 `git lfs status` 验证实际追踪状态；
- 如果大文件已经进入普通 Git 历史，不得只新增 `.gitattributes`，应提审后使用历史迁移工具修复。

不得为了规避限制随意压缩或拆分来源不明的文件。若 LFS 配额不足，应排除文件，在 README 中说明获取途径。

### 8. 第二次提审：执行结果

提交前展示：

- 实际新增、修改、删除列表；
- 与方案不同之处及原因；
- 安全扫描、重复检查和完整性检查结果；
- LFS 对象、普通大文件和仓库体积；
- 无法自动核验、需要人工打开检查的文件；
- 建议 commit 信息和 PR 描述。

只有用户明确确认后，才能 `git add` 和 commit。只有用户明确要求时才能 push、创建 PR 或改写历史。

### 9. Commit 与 PR

一个 PR 应围绕一组相关课程资料，避免混入用户原有改动。PR 描述使用 `docs/CONTRIBUTING.md` 模板，并补充：

- Agent 名称及其执行范围；
- 人工确认过的节点；
- 被排除文件及原因；
- LFS 使用和配额说明；
- 尚需维护者人工检查的内容。

推送前再次确认没有凭证、模型、数据集、临时规划文件和未追踪垃圾。创建 PR 后向用户返回分支、commit、PR 链接以及未完成事项。

## 必须暂停并提审的节点

出现以下任一情况时，Agent 不得自行猜测：

- 无法确定学期、课程或资料类型；
- 需要覆盖、合并或删除非空文件；
- 发现可能的真实凭证或高风险个人信息；
- 资料可能受版权、保密或学术诚信限制；
- 文件超过 50 MiB、需要新增 LFS 规则或可能消耗大量配额；
- 需要安装软件、联网下载、写入仓库外位置；
- 需要 commit、push、创建 PR、强制推送或改写历史；
- 自动检查无法覆盖 PDF 扫描页、图片水印或视频内容。

## 完成标准

工作结束时，源资料保持不变；仓库目录符合规范；项目说明可用；没有凭证、模型、公开数据集和垃圾产物；大文件追踪正确；用户已看过最终差异并明确授权提交或 PR 操作。
