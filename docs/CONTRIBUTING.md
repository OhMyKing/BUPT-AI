# 贡献与 PR 规范

## 目录与命名

1. 使用 `YY_fall` 或 `YY_spring` 作为一级目录。
2. 二级目录使用教务系统中的完整课程中文名。
3. 三级目录按实际内容使用 `教材`、`课件`、`作业`、`实验`、`考题`、`课设`、`论文`、`复习资料` 等；不要创建空分类。
4. 文件名应包含足以区分资源的信息，如年份、考试类型、作业序号和版本。允许保留或使用真实姓名、学号、用户名等课程归档信息。

## 向已有目录贡献不同版本

向已有的课程资源类型目录追加内容时，可以在该目录下使用自己的姓名、用户名或项目名创建子目录，再将资源放入其中。这样可以并列保留不同贡献者的作业、实验实现或项目版本，避免文件重名和相互覆盖。

例如，向 `24_spring/机器智能/作业` 贡献新资源时，可以使用以下结构：

```text
24_spring/机器智能/作业/
└── 王殿云/
    ├── README.md
    ├── 作业报告.pdf
    └── src/
```

子目录也可以命名为 GitHub 用户名或明确的项目名。请确保同一贡献中的命名一致，并在 README 或 PR 描述中说明资源来源、作者和版本差异。若现有目录已经按贡献者或项目划分，请沿用其结构。

## 禁止提交

- API Key、Token、密码、私钥、Cookie、数据库连接凭证；
- `.env` 等真实配置文件，应改为 `.env.example`；
- 训练模型与权重，如 `.pt`、`.pth`、`.ckpt`、`.h5`、`.onnx`、`.safetensors`；
- 可公开下载的数据集及其镜像，应在 README 中提供官方来源、版本与预处理方法；
- `node_modules`、虚拟环境、IDE 缓存、编译产物、日志、临时文件和嵌套 Git 仓库；
- 身份证件、签字页、查重报告、成绩评定表等高风险或无必要公开的敏感材料；
- 无明确授权传播的商业资源。

## 代码项目要求

每个课设或独立代码项目至少包含 README，说明：

- 项目目标与课程背景；
- 目录结构和主要模块；
- 环境与依赖安装方式；
- 配置、运行和复现实验的方法；
- 已移除的数据集、模型或服务及其获取方式；
- 演示材料的位置。

任何凭证都必须由环境变量或本地配置注入。README 示例值只能使用显然无效的占位符。

## PR 自检清单

- [ ] 资源位于正确学期、课程和类型目录；
- [ ] 已检查重复文件和多版本，PR 中解释保留理由；
- [ ] 已清除真实凭证和高风险敏感材料；
- [ ] 未包含模型、公开数据集、依赖和构建产物；
- [ ] 新增项目包含可执行的 README；
- [ ] 大型演示媒体使用 Git LFS；
- [ ] PR 描述列出来源、课程、学期、资源类型和版权情况。
